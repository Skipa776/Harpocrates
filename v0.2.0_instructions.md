# Harpocrates v0.2.0: ML Overhaul Master Design Document

## Part 1: Telemetry Constraints & Pipeline Architecture

### The AppSec Reality of False Negative Feedback Loops
In traditional machine learning, models are iteratively improved by collecting real-world failures from users. In the context of AppSec and secret scanning, relying on users to report False Negatives (missed secrets) is fundamentally unworkable and highly dangerous.

* **The "Live Credential" Paradox:** If Harpocrates fails to detect a valid, highly sensitive production database password, the only way a user can report this failure is by providing the context and the token itself. Submitting a live credential to a public GitHub Issue or support ticket is a catastrophic security breach. 
* **The Sanitization Trap:** Users might attempt to redact or sanitize the secret before submitting it (e.g., changing `password123!` to `REDACTED`). By replacing the actual token with a dummy string, the user destroys the exact entropy, structural anomaly, or contextual formatting that caused the ML model to fail. The reported data becomes entirely useless for retraining.
* **The Zero-Trust Alternative:** We must definitively block any UI/UX prompts that encourage reporting missed secrets. Instead, we rely exclusively on:
  * **Offensive Security Corpora:** Sourcing positive data from known, sanitized leaks (CredData, Betterleaks).
  * **Synthetic Generation:** Using LLM-driven pipelines to generate localized, safe positive class data.
  * **False Positive Reporting:** We will allow users to report False Positives (e.g., "Your model flagged this harmless CSS hex code") to help tune our XGBoost precision model.

### Architectural Necessity of the Pipeline Cascade
Wiring the `--ml` flag as an additive cascade (rather than an override) is a mandatory architectural requirement for optimizing secret detection.

* **The Fast Path (Regex) is Non-Negotiable:** Machine Learning models are computationally heavier than standard regular expressions. Pre-commit hooks operate under extreme latency constraints (<15ms per file). Regex instantly eliminates 90% of standard, high-confidence secrets at $O(n)$ complexity. 
* **Avoiding Model "Overthinking":** ML models can occasionally "overthink" and misclassify highly rigid, obvious tokens that a simple regex would catch deterministically.
* **Defense in Depth (The Cascade Architecture):**
  1. **Regex Engine (The Net):** Catches standard, highly structured credentials.
  2. **Entropy Gate (The Filter):** Drops obvious plaintext English prose.
  3. **Stage A - LightGBM (The Anomaly Hunter / Recall):** Scans the remaining unstructured or low-entropy tokens looking for contextual clues.
  4. **Stage B - XGBoost (The Verifier / Precision):** Aggressively filters out False Positives using negative-class training.

## Part 2: ML Data Generation Strategy - Targeting AI-Generated Secrets

Relying purely on entropy and regex fails when dealing with AI-generated code. LLMs frequently leave behind low-entropy, syntactically valid placeholders (e.g., `sk-YOUR_API_KEY_HERE`, `password = "test"`). 

### 1. The "AI Leftover" Paradigm
AI coding assistants produce code that looks structurally perfect but contains semantic anomalies in the secret values:
* **The Copilot Effect:** LLMs generate standard dummy values (`123456789`, `changeme`).
* **The Template Effect:** LLMs generate instructional strings (`YOUR_AWS_KEY`).
* **The Mocking Effect:** LLMs generate pseudo-random strings with zero real-world entropy (`00000000-0000-0000-0000-000000000000`).

### 2. Synthetic Context Generation Pipeline
We will use local open-source LLMs (e.g., Llama-3, Mistral) to generate positive training data by forcing them to hardcode credentials in various contexts (e.g., "Write a Python script connecting to Postgres and hardcode the credentials"). This provides the exact `context_before`, `context_after`, and `line_content` that developers see when accepting autocomplete suggestions.

### 3. Context-Aware Feature Engineering
We will extract the following contextual features for the ML Pipeline:
1.  **`variable_name_risk_score`**: Score the assignment target against a dictionary of high-risk terms (`secret`, `key`, `token`, `pwd`).
2.  **`file_path_risk`**: Assess risk based on file extension and naming (`.env.example`, `test_*.py`).
3.  **`context_keyword_density`**: Scan surrounding context for connection string keywords (`jdbc:`, `Bearer`, `Authorization:`).
4.  **`token_ai_placeholder_match`**: Flag tokens containing typical AI placeholder substrings (`YOUR_`, `_HERE`, `dummy`).
5.  **`is_in_test_directory`**: Identify mock data in `/tests/` or `/specs/` directories.

### 4. Negative Data Sourcing (The False Positive Sandbox)
To train Stage B (Precision), we will source negative data from:
* **Lock files:** `yarn.lock`, `package-lock.json` (SHA hashes).
* **CSS/SVG:** Hex colors, SVG path data.
* **Commit Hashes:** Hardcoded git SHAs.
* **Public Keys:** GPG public blocks or base64 encoded certificates.

## Part 3: Architectural Evaluation & ONNX Implementation

Shifting our inference engine from native LightGBM/XGBoost Python libraries to **ONNX (Open Neural Network Exchange)** is critical for meeting our pre-commit hook constraints.

### Reasoning: Why ONNX?
* **Dependency Bloat (<4MB Constraint):** The combined footprint of `scikit-learn`, `lightgbm`, and `xgboost` exceeds 200MB. ONNX allows us to drop these dependencies, requiring only the lightweight `onnxruntime` and the `.onnx` graph files (<500KB).
* **Single-Row Inference Latency (<15ms Constraint):** The Python overhead for converting single strings into native `DMatrix` or `Dataset` formats introduces a 10–25ms latency penalty per inference. The ONNX Runtime C-backend executes single-row tree traversals in <1ms.
* **Pipeline Uniformity:** ONNX allows us to load both LightGBM and XGBoost models into a single uniform runtime session, saving memory overhead.

### Implementation Steps

1. **Train in Native Frameworks (Offline):** Use standard Python environments to train models using the synthetic context data, retaining hyperparameter tuning and visualization tools.
2. **Convert to ONNX Graph:** Export models using `onnxmltools.convert_lightgbm` and `onnxmltools.convert_xgboost`. Define `initial_types` as floating-point tensors matching the contextual feature count.
3. **Decouple the Pre-commit Environment:** Remove `xgboost`, `lightgbm`, and `scikit-learn` from `requirements.txt`. Add `onnxruntime` (or `onnxruntime-lite`) and `numpy`.
4. **Refactor the Python Inference Class:** Modify CLI wiring to instantiate an `onnxruntime.InferenceSession`, converting Python lists directly to numpy float32 arrays to eliminate pandas overhead.
5. **AppSec Payload Verification:** Integrate a cryptographic hash check to verify the `.onnx` file's SHA-256 hash against a hardcoded constant before loading, preventing supply-chain attacks.
